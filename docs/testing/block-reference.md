## Engine model

- The engine is **`executeGraph`** (`interpreter.ts`): a pure walker over the workflow graph. It takes an **injectable `executeBlock`** and a set of **hooks** (`terminate`, `clarificationExit`, `failureExit`, `notify`, Jira/Slack side effects), so the graph logic is testable in isolation from real integrations.
- Blocks are dispatched three ways: **control blocks** (`branch`, `loop`, `terminate`) are walked **inline** by the interpreter and never go through `executeBlock`; **registry blocks** are looked up in `BLOCK_EXECUTORS` (`agent.ts:45-57`) and run their own `blocks/*.ts` `execute`; **inline switch blocks** (`open_pr`, `run_pre_pr_checks`, `update_ticket_status`, `send_slack_message`, and the agent phases) are handled by a `case` in `agent.ts`'s `executeBlock` after the registry lookup misses.
- **Triggers** are never executed. `agent.ts` seeds the trigger's output into `steps[triggerId]` before the walk starts (`interpreter.ts:106`); `triggerTypeFor()` maps the entry kind to the trigger type (`agent.ts:64-65`).
- **Validation is deterministic.** Param bounds (zod, per-block `.strict()` schemas) and graph rules (`validateWorkflowGraph`, `schema.ts`) are enforced at **save** time, inside the store rather than only in the routes, and are exhaustively covered by the repo's **1536-test worker suite** (132 files, 0 failing, as of 2026-07-15) (`schema.test.ts`, `schema.edge.test.ts`, `interpreter.test.ts`, `interpreter.edge.test.ts`, `golden-runs.edge.test.ts`, `io-blocks.edge.test.ts`, and the per-block `*.test.ts` files). "Live-exercised" below means the block runs on a real agent run, either the Tier 0 pipeline or a purpose-built definition, not just in the deterministic suite.

## Necessity verdicts

Grouped by verdict so you can see at a glance which blocks may be unnecessary. Tally: **core 13, niche 13, redundant 1** = 27. The audit's fourth verdict, **removable 1** (`arthur_trace`), has since been acted on: the block was removed in `2d0912a`, which is why the tally is 27 and not 28.

### core (13)

- **trigger_ticket_ai** - sole non-webhook entrypoint and the only trigger with a default fallback, so a run always has a runnable plan; backbone of the Tier 0 pipeline.
- **prepare_workspace** - sole provider of the run sandbox; auto-injected when omitted (`definition-step.ts:31-33`); every agent/checks/publish block hard-fails "no workspace" without it.
- **planning_agent** - first agent phase; its plan (`ctx.researchPlanMarkdown`) seeds implementation and review.
- **implementation_agent** - the only block that produces committed code changes; open_pr publishes them and pre-PR checks/PR use its model+kind.
- **branch** - the only conditional-routing primitive; no other block splits control flow.
- **loop** - the sole re-entrant/retry construct; the entire cycle-validation subsystem is built around it.
- **terminate** - the only block that lets an author explicitly choose a terminal disposition (waiting_for_human / failed / skipped / done).
- **send_plan_approval** - sole producer of `approval_requests` rows and the only park-for-approval mechanism (core-but-conditional on the HITL feature being wired).
- **post_ticket_comment** - the only workflow-authorable way to post an arbitrary comment on the ticket mid-run.
- **update_ticket_status** - the only in-engine way to land a ticket in the AI Review column (the engine auto-moves only to backlog); closing step of the default workflow.
- **open_pr** - sole block that publishes the workspace and opens the PR; terminal action of the default ticket to PR pipeline; the only one that records `prForTelemetry`.
- **run_pre_pr_checks** - the pre-PR gate baked into the default workflow; the only block that can hard-block a run on check failures with auto-fix cycles.
- **call_llm** - the only block that invokes an LLM in-process (no sandbox, no agent CLI); a cheap classify/route/summarize step, non-redundant though optional/advanced.

### niche (13)

- **review_agent** - optional quality gate: core when the pipeline enables review (`includeReview`), otherwise skipped; its only product (`feedback`) is consumed only if wired.
- **fix_agent** - integration-dependent; dead outside PR flows (only `trigger_pr_checks_failed` / `trigger_pr_review` entries populate its inputs).
- **generic_agent** - power-user escape hatch used by no default pipeline; runs a full coding agent in the sandbox (distinct from call_llm).
- **human_question** - no side effects of its own; overlaps the engine's needs_human_input contract; useful only as an explicit authoring primitive.
- **trigger_plan_approved** - required only for the send_plan_approval to human-approval to resume loop; dead unless the approval HITL is wired.
- **trigger_pr_created** - real capability but PR-flow-gated; dead unless VCS webhooks are wired and a PR-triggered definition is enabled.
- **trigger_pr_checks_failed** - PR-flow-gated; distinct `failedChecks` payload so not redundant.
- **trigger_pr_review** - PR-flow-gated; closest to overlapping trigger_pr_checks_failed (identical downstream walk) but distinct review payload.
- **post_pr_comment** - the only block that writes a comment on a PR; dead unless a PR-oriented workflow is authored.
- **fetch_pr_context** - read-side of PR/review workflows; duplicates the legacy inline path but is the only way to refresh PR context mid-graph.
- **run_checks** - report-only check runner; its no-commands path partially duplicates run_pre_pr_checks; distinctive value is `commands` mode (branchable pass/fail).
- **arthur_injection_check** - fully wired and well-tested but inert unless Arthur is provisioned AND a branch reacts to `status:"flagged"`; report-only.
- **send_slack_message** - misleadingly named; really the fixed "pr_ready" notification step; silent no-op (skipped) before a successful publication and fully dead without the Slack env vars.

### redundant (1)

- **finalize_workspace** - the publish half is a near-verbatim duplicate of the inline `open_pr` case (`agent.ts:1093-1140`); the default pipeline uses open_pr, and its sole value-add is the `requiredChecks` gate (otherwise expressible as a branch on run_checks output). Carries a latent divergence: it pushes with the run-default kind/model for the push-fix retry instead of the implementation agent's.

### removable (1) - acted on, block removed

- **arthur_trace** - **removed in `2d0912a`** (2026-07-14 13:36), after this audit recommended it. The verdict at the time: a pure marker whose only effect (renaming the Arthur observability task) was achieved by a **static pre-scan** that did not even require the node to be reached; its `execute()` was a literal no-op. The same override could live as a prepare_workspace param or a run-level setting, and its one real effect was untested end-to-end. Kept here as the record of why the block is gone; see the `arthur_trace` entry under Group 9 for what it did.

### Concrete dead-config and overlap findings

These are the specific "this config does nothing" and "these two blocks overlap" facts the audit surfaced:

- **trigger_plan_approved.source** - enum-of-one (`'dashboard'`, `schema.ts:52`), **never read at runtime** (grep: zero reads). Pure declarative placeholder; removable dead config.
- **trigger_pr_created.providers** - **never read at runtime** (grep: zero reads). Provider scoping is done only per webhook endpoint, not by this param. Declarative-only / dead.
- **trigger_pr_created.onlyWorkflowOwned** - read **only** to emit a warn log when `false` and explicitly **not honored** (`dispatch-trigger.ts:48-53`). The real, always-on filter is `ticketKeyFromBranch` (bot-branch `blazebot/` prefix, `branch-prefix.ts:9-12`). Warn-only / semi-dead.
- **trigger_pr_checks_failed.providers** - same as above; never enforced at runtime.
- **trigger_pr_review.on** - **never read at runtime** (grep: zero reads). Both `changes_requested` and `commented` reviews normalize into `trigger_pr_review` regardless; narrowing must be done with a downstream branch on `steps.<trigger>.output.review.state`. Also `providers:['gitlab']` is a no-op (no GitLab review mapping). Dead / removable.
- **run_checks vs run_pre_pr_checks** - run_checks' no-commands path duplicates run_pre_pr_checks but **report-only** (`maxFixCycles=0`, returns `next` + an `ok` flag instead of gating `failed`). run_checks' distinctive value is the `commands` mode (arbitrary per-repo commands a graph author branches on).
- **call_llm vs generic_agent** - both go prompt to structured output, but **not redundant**: generic_agent spins up the full coding agent inside the sandbox (can run tools on the workspace) whereas call_llm is a stateless single in-process LLM call with no sandbox.
- **finalize_workspace vs open_pr** - see the redundant verdict above.
- **update_ticket_status.target** - typed as a free string (`min(1).max(100)`) but only `'backlog'` is special-cased; **every other value** (including `'done'`, capitalized `'Backlog'`, typos) silently maps to `ai_review` (`ticket-move-target.ts:5`). A foot-gun; a 2-value enum would be safer. The editor hides this by offering only two values.
- **ctx.moveTargets.aiReview** - dead (`agent.ts:775`): never read; only `ctx.moveTargets.backlog` is consumed (by `send-plan-approval.ts:126`).

## What each block needs to run live

**There are no live-coverage gaps left.** All 27 current block types were exercised on real runs during the 2026-07-14 campaign (`e2e-test-report.md` section 2 lists the vehicle and evidence per block). This section was originally the gap list; it is kept as the standing reference for **what infrastructure each block needs before it can run live**, which is still accurate and is what a future block author or a fresh environment has to satisfy. Each group notes how the campaign satisfied it.

**Need Arthur provisioned** (`GENAI_ENGINE_API_KEY` + `GENAI_ENGINE_TRACE_ENDPOINT`, both optional in `env.ts:88-89`, plus prepare_workspace ensuring the task):

- **arthur_injection_check** - inert unless Arthur is wired; additionally needs `ctx.arthur.taskId` (only set when prepare_workspace ran with Arthur configured). Closed by the campaign's Arthur definition: ran a real prompt-injection scan, `status: ok`.

**Need a real PR / CI / review webhook** (a `pr_trigger` entry or a published workspace). Closed by opening real bot PRs (#292/#293/#294) on the demo repo and firing real events at them, after the case-sensitive webhook bug (7a) was fixed:

- **fix_agent** - only runs on `trigger_pr_checks_failed` / `trigger_pr_review` entries; its `pr_trigger` branch and `ctx.repositoryContexts` never populate in the standard ticket to PR pipeline.
- **trigger_pr_created**, **trigger_pr_checks_failed**, **trigger_pr_review** - need VCS webhooks wired and a PR-triggered definition enabled.
- **post_pr_comment** - needs a published workspace (`ctx.publication`) or a `pr_trigger` entry to have a PR in scope.
- **fetch_pr_context** - needs `selectedRepositories` with a workflow-owned PR, or a `pr_trigger` entry; returns empty context otherwise.
- **finalize_workspace** - the default pipeline publishes via open_pr, so finalize's real push path needs a definition that places it. Closed via `implementation_agent -> finalize_workspace` (def 13), which opened PR #292; note that it needs a **committed** diff (see `e2e-findings.md` 7c).

**Need the HITL approval infra** (dashboard approve/reject routes + a stored studio definition). Closed by the campaign's approval definition (def 11):

- **send_plan_approval** - needs `ctx.definitionId` (a persisted studio definition) plus the dashboard approve/reject routes and the `trigger_plan_approved` chain deployed. Parks with `awaiting_approval` by design, which is its success state.
- **trigger_plan_approved** - only dispatched by the approvals API on grant; no default fallback, so it needs the approval loop wired. Fired by approving the parked plan.

**Feasible via a ticket-triggered custom definition** (these run on the standard ticket sandbox, no external webhook / Arthur / approval infra needed, only the baseline Jira / LLM integrations). Closed by the LLM, loop, HITL and stress definitions:

- **branch**, **loop**, **terminate** - control primitives; a custom ticket definition with these nodes exercises them live (terminate's real Jira-move / Slack-notify hooks additionally need those integrations, but the block itself runs).
- **human_question** - produces needs_human_input; the engine posts the questions to the ticket. Runs live on any ticket definition (Jira baseline). Parks the ticket by design, which is its success state.
- **post_ticket_comment** - runs live on any ticket definition (Jira baseline).
- **run_checks** - runs in the sandbox on any ticket definition.
- **call_llm** - runs in-process on any ticket definition (needs an LLM API key, which is baseline).
- **generic_agent** - runs the coding agent in the sandbox on any ticket definition (baseline sandbox).

## Blocks by group

### Group 1 - Triggers

#### trigger_ticket_ai

- **What it does:** The default, primary run entrypoint. Fires when a Jira ticket enters the AI column (via the Jira webhook or cron poller). Every run that is not a PR webhook or a plan approval enters here.
- **How it works:** Zod params are an empty strict object (`schema.ts:43-45`); spec is category `trigger`, single `out` port, no failure port (`workflow-graph.ts:15`). Never executed through `executeBlock`; `agent.ts` seeds its output into `steps[triggerId]` before the walk (`interpreter.ts:106`) and `triggerTypeFor()` maps entry kind `ticket` to `trigger_ticket_ai` (`agent.ts:65`). Dispatch is `jira.post.ts` to `dispatchTicket` (or the cron poller), not `dispatch-trigger.ts`. Definition resolution `loadWorkflowDefinition()` to `loadWorkflowDefinitionFor('trigger_ticket_ai')` (`definition-step.ts:157-159`). **Unique:** the only trigger with a built-in default fallback: when no valid stored definition exists, it falls back to `defaultWorkflowDefinition()` (`definition-step.ts:108-110,117,127,147`). `normalizeDefinitionForExecution` splices a virtual prepare_workspace after it when missing (`definition-step.ts:27-70`).
- **Params:** none (strict `{}`, `schema.ts:33,44`; `BLOCK_PARAM_KEYS` empty, `workflow-graph.ts:61`).
- **Output shape:** `{ status:'fired', ticketKey }` (`agent.ts:110`), stored under the trigger node id so branch conditions can read `steps.<triggerId>.output.*`.
- **Failure / missing input:** Params can never be missing. At save, unknown keys strict-reject (400). `validateWorkflowGraph` forbids >1 ticket trigger and any incoming edge (`schema.ts:419-423,482-486`). At runtime the trigger cannot fail; if the ticket is not actually in the AI column, `fetchAndValidateTicket` returns null and the run silently returns (`agent.ts:124-127`). The default fallback guarantees a runnable plan even with zero stored definitions.
- **Test coverage:** `schema.test.ts` strict-reject + duplicate/unknown-port/no-incoming rules (177-195, 288-349); `loader-triggers.edge.test.ts` default-fallback + virtual prepare_workspace injection (248-283); `golden-runs.edge.test.ts` entry-id resolution (453-455) + full linear pipeline. No missing-param case exists.

#### trigger_plan_approved

- **What it does:** Resumes a run after a human approves a plan an upstream send_plan_approval parked. Re-enters the graph carrying the approved plan so the run skips planning and goes straight to implementation.
- **How it works:** Params `{ source: literal 'dashboard', default 'dashboard' }` strict (`schema.ts:47-55`). `triggerTypeFor()` maps entry kind `plan_approved` (`agent.ts:64`). Dispatch is `approvals/dispatch.ts` `dispatchPlanApproved()` (called from the approvals API on grant): looks up `getEnabledWorkflowDefinitionForTrigger(db,'trigger_plan_approved')` (`dispatch.ts:52`), runs the claim/capacity dance, then `start()`s with `kind:'plan_approved'` carrying `approvedPlan` + `approver` (`dispatch.ts:99-113`). `agent.ts` seeds `ctx.researchPlanMarkdown` from `entry.approvedPlan.markdown` (`agent.ts:770`). **`source` is never read at runtime** (grep: zero reads): a pure declarative placeholder with one legal value.
- **Params:** `source` enum-of-one `'dashboard'` (default, `schema.ts:52`; `BLOCK_PARAM_KEYS ['source']`, `workflow-graph.ts:62`).
- **Output shape:** `{ status:'fired', ticketKey, approvedPlan:<markdown>, approver, approvedAt }` (`agent.ts:101-109`).
- **Failure / missing input:** `source` defaults, so absence never rejects; the literal rejects any other value at save. As a non-ticket trigger it has **no default fallback**: no enabled definition, or a missing/invalid pinned one, means nothing starts (`dispatch.ts:53-56`; `definition-step.ts:120-122,148`; `agent.ts:635-640`). No needs_human_input at the trigger level.
- **Test coverage:** `schema.test.ts` default applied + unknown-key reject (179,194,202-204); `approvals/dispatch.test.ts` full behavior (no_enabled_definition, already-claimed, at-capacity, started+register, claim-release on throw); `golden-runs.edge.test.ts` the second (plan_approved) chain walk + entry-id resolution (362-408,456).

#### trigger_pr_created

- **What it does:** Fires when a PR (GitHub) or MR (GitLab) is opened or reopened on a bot-owned branch, starting an agent run against that PR.
- **How it works:** Params `{ providers: array<'github'|'gitlab'> default ['github','gitlab'], onlyWorkflowOwned: bool default true }` (`schema.ts:57-68`). `normalizeGitHubEvent` maps pull_request opened/reopened (`trigger-events.ts:38-44`); `normalizeGitLabEvent` maps MR open/reopen (`trigger-events.ts:107-129`). Webhook route calls `dispatchTriggerEvent` (`github.post.ts:62` / `gitlab.post.ts:48`); dispatch resolves the enabled definition (`dispatch-trigger.ts:30`) and gates on `ticketKeyFromBranch(headRef)`, the real always-on "workflow-owned" filter: a non-bot branch returns `ignored_not_workflow_owned` and the post-PR gate runs instead (`dispatch-trigger.ts:36-43`). Then `claimTicketRun` + `start` with `kind:'pr_trigger'` (`dispatch-trigger.ts:64-79`). **`providers` is never read at runtime** (grep: zero reads); provider scoping is only per webhook endpoint. **`onlyWorkflowOwned` is read only to warn-log when false and is explicitly not honored** (`dispatch-trigger.ts:48-53`).
- **Params:** `providers` (defaulted, declarative-only/never enforced), `onlyWorkflowOwned` (defaulted, warn-only) (`schema.ts:57-68`; `workflow-graph.ts:63`).
- **Output shape:** `{ status:'fired', ticketKey, provider, repoPath, prNumber, prUrl, headRef, headSha, baseRef, title, author, isDraft }` (`agent.ts:70-84`).
- **Failure / missing input:** Both params defaulted; unknown key strict-rejects. Under-specified PR fields fill with `''` / `'unknown'` / defaults (`trigger-events.ts:169-182`). Only opened/reopened dispatch; synchronize etc. fall through to the gate. Non-bot branch to `ignored_not_workflow_owned`. `at_capacity`/error return 503 so the provider redelivers.
- **Test coverage:** `trigger-events.test.ts` opened/reopened/synchronize/draft + GitLab (23-72,260-281); `dispatch-trigger.test.ts` no_definition/non-bot-ignore/coalesce/at_capacity/started (104-201); `loader-triggers.edge.test.ts` dispatch area (360-449) + github route 401/malformed/reopened/gate-skip (666-752); `gitlab.post.test.ts`; `schema.test.ts` defaults (205-208). **Gap:** no test asserts `providers`/`onlyWorkflowOwned` are ignored (there is nothing to filter).

#### trigger_pr_checks_failed

- **What it does:** Fires when CI checks (GitHub check_run) or a pipeline (GitLab) fail on a bot-owned PR, kicking off an automated fix run.
- **How it works:** Params `{ providers: array default ['github','gitlab'] }` (`schema.ts:70-78`), no `on`. GitHub: check_run action=completed with conclusion in {failure,timed_out}, excluding the bot's own gate checks by exact name or `blazebot / ` prefix (anti-loop), taking the first attached PR (`trigger-events.ts:46-78`). GitLab: Pipeline Hook status=failed, mapping failed builds, falling back to a single `pipeline` entry (`trigger-events.ts:131-164`). Same `dispatchTriggerEvent` path and always-on bot-branch gate as trigger_pr_created. Carries `failedChecks` into the output.
- **Params:** `providers` only (defaulted, never enforced at runtime) (`schema.ts:75`; `workflow-graph.ts:64`).
- **Output shape:** PR payload **plus** `failedChecks:[{ name, conclusion, detailsUrl? }]` (`agent.ts:85-91`).
- **Failure / missing input:** `providers` defaulted; unknown key rejects at save. Anti-loop guard drops the bot's own gate checks; non-failure conclusions and a failed check with zero attached PRs return null. Under-specified fields default (`title=''`, `author='unknown'`). Empty/non-bot headRef to `ignored_not_workflow_owned`. Same 503-redelivery.
- **Test coverage:** `trigger-events.test.ts` failure mapping, gate anti-loop exact+prefix, success ignored, no-PR ignored, GitLab pipeline variants (74-174,283-323); `golden-runs.edge.test.ts` runs the shared fix pipeline to completion, two PR triggers keep separate outEdges converging on fetch-context (410-438,457); `github.post.test.ts` check_run dispatch skips gate (743-751); `schema.test.ts` defaults (209-211).

#### trigger_pr_review

- **What it does:** Fires when a human submits a PR review that requests changes (or comments) on a bot-owned PR, starting a run to address the feedback.
- **How it works:** Params `{ providers: array default ['github'], on: enum('changes_requested'|'commented') default 'changes_requested' }` (`schema.ts:80-91`). `normalizeGitHubEvent` maps pull_request_review action=submitted with review.state in {changes_requested,commented}, skipping the bot's own reviews when `VCS_BOT_LOGIN` matches (`trigger-events.ts:80-98`). **No GitLab review mapping.** Same dispatch path + bot-branch gate. Carries `review {state,author,body}`. **`on` is never read at runtime** (grep: zero reads): both states normalize into trigger_pr_review regardless; any narrowing needs a downstream branch on `steps.<trigger>.output.review.state`.
- **Params:** `providers` (defaulted; `'gitlab'` would be a no-op), `on` (defaulted, never enforced) (`schema.ts:85-88`; `workflow-graph.ts:65`).
- **Output shape:** PR payload **plus** `review:{ state, author, body }` (`agent.ts:92-98`).

#### arthur_injection_check

- **What it does:** Report-only prompt-injection screen. Runs Arthur's `validate_prompt` against either a referenced upstream block's output or the ticket description+comments, and emits a single `kind:"next"` output carrying status `"ok"` | `"flagged"` | `"skipped"` so a graph can branch on it. Observational only: never blocks or fails the run.
- **How it works:** execute at `arthur-injection-check.ts:41`. (1) Guards: if `!GENAI_ENGINE_API_KEY || !GENAI_ENGINE_TRACE_ENDPOINT` to `{status:"skipped", reason:"arthur_not_configured"}`; if `!ctx.arthur.taskId` to skipped/"arthur_task_missing". `ctx.arthur.taskId` is populated only by prepare_workspace's ensure-Arthur-task step (`prepare-workspace.ts:49-71`, env-gated), so this block yields a real result only on runs where prepare_workspace ran first **and** Arthur is configured. (2) Content selection: if `contentFromStep` non-empty, `steps[contentFromStep]`; missing that step to skipped/`no output from block "X"`, else `content = JSON.stringify(source.output)`; otherwise `content = [ticket.description, ...comments].filter(Boolean).join("\n\n")`. (3) Calls `blockArthurValidatePromptStep` (maxRetries=0): re-checks env and throws "Arthur is not configured", builds `ArthurClient.fromTraceEndpoint`, calls `client.validatePrompt(taskId, content)` which POSTs `/api/v2/tasks/{id}/validate_prompt`, requires array `rule_results` else throws, `ok = every finding.result==="Pass"`. (4) Result mapping to status ok|flagged with mapped findings. (5) catch: **any** error (500, unexpected shape, network, the internal env throw) to `{status:"skipped", reason: message.slice(0,300)}`. Registered in dispatch map at `agent.ts:35,57`.
- **Params:** `contentFromStep?: string` (`z.string().trim().min(1).optional()`, strict) (`arthur-injection-check.ts:4-8`; schema `schema.ts:167-173`; `BLOCK_PARAM_KEYS ['contentFromStep']`, `workflow-graph.ts:84`). Only param, optional, so no required param.
- **Output shape:** `{ status:"ok", findings:[] }` | `{ status:"flagged", findings:[{rule,result,details?}] }` | `{ status:"skipped", reason }`. Always `{ kind:"next", output }`. Spec category `action`, ports `["out"]`, `allowsFailurePort true`, but execute never returns `kind:"failed"`, so the wirable `failed` port is **unreachable**.
- **Failure / missing input:** No required param, so no missing-required-param path. zod rejects empty-string `contentFromStep` and unknown keys at **save**. All under-specification is a soft runtime **skip**, never a hard fail and never needs_human_input: Arthur unconfigured to skipped/arthur_not_configured; no Arthur task to skipped/arthur_task_missing; contentFromStep points at a block that did not run to skipped/`no output from block "X"`; Arthur API error to skipped/reason. A ticket that under-specifies never trips this block; the run continues on `out` with a skipped status a downstream branch may or may not inspect.
- **Test coverage:** Own unit suite `arthur-injection-check.test.ts` covers all three axes: params accept `{}`/`contentFromStep`, reject `""`/extra (:24-31); referenced-block-missing to skipped (:110-119); ok with ticket content incl. exact validatePrompt args (:57-70), flagged findings (:72-92), contentFromStep uses JSON.stringify (:94-108); not configured (:40-46), no task (:48-55), client error to skipped not thrown (:121-133). `schema.test.ts` covers zod at definition level (:169,190). **Not referenced by any deterministic `*.edge` suite** (grep found zero arthur refs), so the end-to-end path through prepare_workspace to ctx.arthur.taskId to this block is unexercised by the golden/edge harness. The internal env-throw is dead in practice (execute already guards env, catch would downgrade to skipped).

#### arthur_trace (REMOVED in `2d0912a`)

> **This block no longer exists.** It was live-exercised during the 2026-07-14 campaign (`e2e-test-report.md` records it as `ok` on the Arthur definition), and removed later the same day at 13:36 by `2d0912a` ("refactor(worker): drop dead blocks/params..."), acting on this audit's **removable** verdict. `blocks/arthur-trace.ts`, `blocks/arthur-trace.test.ts`, the `agent.ts` pre-scan, and the `WorkflowBlockType` member are all gone; the `taskName` override no longer exists in any form, and the Arthur task now always takes its name from `ticket.identifier`. The description below is the pre-removal record and its `file:line` citations refer to deleted code. It is kept so the removal rationale stays auditable, not as current API. Nothing needs to be done about it.

- **What it did:** Declarative marker with no runtime behavior of its own. Its `taskName` param renamed the run's Arthur observability task; the block's own `execute()` was a pure no-op that returned status "ok".
- **How it worked:** The block's execute did nothing but return `status:"ok"`. The load-bearing effect was a **static pre-scan** in agent.ts, independent of graph execution: before any block ran, `plan.nodes.find(n => n.type==="arthur_trace")?.params` read the **first** arthur_trace node's `taskName`; if a non-empty trimmed string it became `arthurTaskNameOverride` and was written into `ctx.arthur.taskNameOverride`. prepare_workspace's ensure-task step then named the Arthur task via `ctx.arthur.taskNameOverride ?? ctx.ticket.identifier` to `ArthurClient.ensureTaskForTicket` (resolve-or-creates, appends `.1`/`.2` suffixes on re-runs). Key consequences: (a) the block never needed to be reached during execution for its effect to apply; position and edges were irrelevant; (b) only the first arthur_trace node won, extras were silently ignored; (c) executing the block was decorative. This gap between "what the node looks like it does" and "what actually does it" is what made it removable.
- **Params:** `taskName?: string` (`z.string().trim().min(1).max(200).optional()`, strict). Only param, optional.
- **Output shape:** Always `{ kind:"next", output:{ status:"ok" } }`. Spec category `action`, ports `["out"]`, `allowsFailurePort true`, but execute could never fail, so the `failed` port was **unreachable**.
- **Failure / missing input:** `taskName` optional. Absent, or empty/whitespace (rejected at save by min(1)) meant no override, so the Arthur task fell back to `ticket.identifier`; a bare arthur_trace with no taskName was a pure no-op with **zero** observable effect (identical to the block not existing). Never failed, never needs_human_input; only zod max(200)/empty-string/unknown-key rejected at save. Under-specification footgun: two arthur_trace nodes with different taskNames meant the first by plan.nodes order silently won; not validated against.
- **Test coverage (at removal):** Own unit suite `arthur-trace.test.ts` covered only the inert surface: params accept `{}`/`taskName`, reject `""`/extra; execute no-op returns `{kind:"next",output:{status:"ok"}}`. `schema.test.ts` covered zod at definition level. **Critical gap that motivated removal:** the block's only real behavior, the pre-scan copying taskName into `ctx.arthur.taskNameOverride` and its consumption in prepare_workspace, was **not tested anywhere**. So happy-path/failure/missing-input were covered only for the no-op + schema, while the load-bearing override was entirely unverified end-to-end. `2d0912a` deleted the unit suite along with the block.
