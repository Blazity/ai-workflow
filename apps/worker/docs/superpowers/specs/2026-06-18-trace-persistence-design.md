# Persist trace step waterfalls to Postgres

**Date:** 2026-06-18
**Status:** Approved (design)
**Scope:** `apps/worker` only. The dashboard is untouched — the `/api/v1/runs/[runId]` response shape is unchanged.

## Problem

The single-run trace waterfall (the per-step timeline on the Trace screen) is read
live from the Vercel Workflow `world` (`collect-run-detail.ts`), which retains
steps for only ~24h. After that, the route falls back to a *coarse* phase
waterfall synthesized from the `workflow_runs.phases` JSONB
(`run-detail-read.ts` → `phasesToSteps`). So real per-step traces are not durable
and not read from one place: live runs and recent runs come from `world`, old
runs degrade to fake phase pseudo-steps.

Goal: persist the real step waterfall into Postgres and make Postgres the single
read source for finished runs, so trace data lives in one place alongside the
rest of the run telemetry (`workflow_runs`).

## Decisions (resolved during brainstorming)

1. **Hybrid read:** `world` stays the source for in-flight runs (near-real-time,
   ~15s cache); Postgres is the source for finished runs. The final waterfall is
   snapshotted on completion.
2. **Writer = workflow completion path** (`recordRunUsage`, via the agent
   workflow's `recordRunTelemetryStep`). Universal coverage — works on every
   deployment regardless of whether the scheduled cron fires. (The cron writer
   was rejected because `recordRunUsage`'s own comments document that the cron
   does not fire on some deployments.)
3. **Storage = a `steps` JSONB column on `workflow_runs`**, mirroring the
   existing `phases` JSONB. No separate `run_steps` table: traces are always
   read as a whole set per `runId`, never queried cross-run.
4. **Cosmetic last-step normalization:** included (see Known Limitation).
5. **Coarse `phasesToSteps` fallback:** kept, for pre-migration runs with no
   persisted steps. No regression.

## Architecture

### Storage

Add one column to `workflow_runs` (`db/schema.ts`):

```ts
/** Full RunStep[] trace waterfall, captured on completion (workflow-owned). */
steps: jsonb("steps"),
```

The column stores the `RunStep[]` contract verbatim
(`stepId, name, rawName, status, attempt, createdAt, startedAt, completedAt,
startOffsetMs, durationMs, error`) — exactly what the API returns and the trace
screen consumes, so reads require no transformation.

New drizzle migration — the next sequential file (existing migrations run to
`0002`, so this is `drizzle/0003_<name>.sql`), generated by `pnpm db:generate`:

```sql
ALTER TABLE "workflow_runs" ADD COLUMN "steps" jsonb;
```

The migration SQL must be generated and committed before tests run: the test
harness (`db/test-db.ts`) applies the committed `drizzle/*.sql` files to an
in-memory PGlite database, so the column only exists in tests once its migration
file is present.

### Write path (capture on completion)

The agent workflow already records cost/tokens/phases on every exit via the
`recordRunTelemetryStep` `"use step"` in its outer `finally`
(`workflows/agent.ts`). Extend it:

1. Inside the step, fetch the run + its steps from the world:
   `getWorld().runs.get(runId, { resolveData: "none" })` and
   `getWorld().steps.list({ runId, resolveData: "none", pagination: { limit: 100 } })`.
2. Map them to `RunStep[]` using the **shared mapper** (below).
3. Pass the mapped `steps` into `recordRunUsage` as a new `RunUsage.steps` field.

Best-effort: the whole capture is wrapped so a `world` failure leaves
`steps = null` — cost/status still record exactly as today. The step keeps
`maxRetries = 0` and the caller keeps swallowing errors; telemetry must never
retry or fail the run.

`recordRunUsage` (`lib/telemetry/run-telemetry.ts`):
- `RunUsage` gains `steps: RunStep[] | null`.
- Insert sets `steps`.
- `onConflictDoUpdate` uses `coalesce(excluded.steps, "workflow_runs"."steps")`
  (the existing `keepIfNull` helper) so a second exit-path write or a failed
  capture never erases an already-captured waterfall.
- The cron writer (`upsertRunSnapshots`) does **not** touch `steps` — disjoint
  columns, unchanged.

### Capture helper (reuse `collectRunDetail`)

Rather than extracting a separate step mapper, the capture path **reuses
`collectRunDetail` wholesale** — it already turns a `world` source + `runId`
into `{ run, steps }` with the exact mapping the live read uses, so the
persisted waterfall is byte-identical for free. A thin best-effort wrapper
(`collect-run-detail.ts`) isolates the discard-header + swallow-errors logic so
the capture never throws inside the telemetry step:

```ts
/** Capture just the step waterfall for a run; null on any world failure. */
export async function captureRunStepsBestEffort(
  world: RunDetailSource,
  runId: string,
): Promise<RunStep[] | null>
```

`collectRunDetail` is otherwise unchanged.

### Read path

`routes/api/v1/runs/[runId].get.ts`:

```
row = fetchRunDetailFromDb(runId)            # one query: status, steps, refs
if (row has real persisted steps)            # finished run
    return row                               #   → Postgres (single source)
try
    detail = collectRunDetail(world) + DB refs   # in-flight run
    return detail                            #   → world (live)
catch (world unavailable / aged out)
    return row ?? EMPTY                       #   → coarse phase fallback / empty
```

Finished runs never call `world`. In-flight runs use `world` exactly as today.

### Backward compatibility

`db/queries/run-detail-read.ts` → `fetchRunDetailFromDb`:
- If `row.steps` is present (non-empty array) → return those **real** steps and
  signal `hasRealSteps = true`.
- Else → fall back to the existing `phasesToSteps(row.phases, base)` coarse
  synthesis (`hasRealSteps = false`).

Result: new finished runs render the real waterfall from Postgres;
pre-migration runs with no `steps` behave exactly as today (world if <24h, else
coarse phases).

## Known limitation

The telemetry step captures the waterfall while it is itself running, so the
persisted trace shows *that one bookkeeping step* (`recordRunTelemetryStep`,
not user-meaningful agent work) as not-completed. Mitigation (included): the
read path normalizes any non-terminal step in a **finished** run's persisted
waterfall to `completed`, stamping `completedAt`/`durationMs` from run
completion. ~3 lines, cosmetic only.

## Testing (TDD)

- `run-telemetry.test.ts`: `recordRunUsage` persists `steps` on insert;
  a re-write with `steps: null` does not clobber a previously-captured waterfall.
- `run-detail-read.test.ts`: `fetchRunDetailFromDb` returns real `steps` when
  present and prefers them over `phasesToSteps`; still synthesizes phases when
  `steps` is null.
- `collect-run-detail.test.ts`: the extracted `mapWorldSteps` produces the same
  output the inline mapping did (no behavior change to the live read).
- Route test (`runs/[runId]`): a terminal run with persisted steps reads from
  Postgres without calling `world`; an in-flight run reads from `world`.

## Out of scope

- Live in-flight traces remain `world`-sourced (no per-step persistence while a
  run is mid-flight).
- The other non-Postgres dashboard stats (live run progress/ETA, eval-health
  stub, Arthur evals/prompts) are unrelated and untouched.
